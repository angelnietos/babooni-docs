## Objetivo

Parametrizar `PrismaMultiService` / `PrismaSingleService` para instanciar el adapter correcto según provider, sin tocar dominios ni repos.

## Acciones

- [ ] Añadir `DatabaseAdapter` interface + factory en `prisma/adapters.ts`:
  - `createPgAdapter(url)`
  - `createMssqlAdapter(url)`
  - `createMysqlAdapter(url)`
  - `createSqliteAdapter(url)`
- [ ] Refactorizar `prisma-multi.service.ts` y `prisma-single.service.ts` para usar factory:
  ```ts
  adapter = createAdapterForProvider(resolveDatabaseProvider(url))(url)
  ```
  Mantener `new PrismaClient({ adapter })`.
- [ ] Mantener `@prisma/adapter-pg` como default dev (localhost:5432) para no romrer setups locales.
- [ ] Error temprano en consola si el provider no tiene adapter en el runtime (lista hardcodeada + docs).
- [ ] Actualizar `prisma.services.spec.ts` para mockear factory y verificar que cada servicio llega a `super()` con el adapter correcto.
- [ ] Añadir smoke en `check:exports-paths` + `nx build base-backend` para garantizar bundle sin providers rotos.

## Criterios

- [ ] Cambio de `DATABASE_PROVIDER=mssql` y `DATABASE_URL=mssql://…` hace que los servicios instancien `@prisma/adapter-mssql`.
- [ ] No quedan imports directos de `PrismaPg` fuera de `libs/base/backend/prisma/`.
- [ ] `nx typecheck base-backend` verde.
- [ ] Tests unitarios servicios verde con cada adapter mockeado.

## Notas

- SqlServer/MariaDB necesitan schema features específicos (F83-C1). Aquí solo se asegura la *conexión parametrizada*; el schema concreto se valida en F83-C1.
- No introducir kappa nuevo mientras no haya package publicado del adapter (si falta, fallar con mensaje accionable).
